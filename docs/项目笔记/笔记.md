### GC
是什么
在单增的过程中是young gen生成的过程，那个断崖式下跌就是GC触发把所有young gen都清理掉了
为什么需要
堆内存是有限制的，满了就会卡住，所以需要把一些用不到的垃圾给清理掉
坑点
在GC过程中会触发STW，整个程序会停止执行，这个过程的快慢往往决定着程序运行速度(这里面的理解核心其实要了解堆，它会扫描堆，在堆里但没人引用的对象会被清理掉，更深一步去了解就是可达性分析了（Reachability Analysis）)

### **哈希函数为什么**O(1)?

经过哈希函数的计算，再进行取模，原来的key会变为在一定范围内的整数，而且这个整数经过这个函数计算并取模后是确定的，因此可以用其作为下标，所以查询速度接近O（1）

### **哈希冲突**？

为了解决多个key经过计算并取模后得到相同的结果，最后会在同一个桶里面，这里采取链表来解决，相同的就继续挂在后面

### **链表太长**？

当数组长度小于64且链表长度大于8时会先采取数组扩容的方法，来缩短链表的长度，如果数组大于等于64且链表长度大于8时，链表还是太长就采取红黑树代替链表

### **线程不安全**？

当扩容数组时，Java7采取头插法进行重新排列链表，多个线程同时操作，这里就有可能出现，节点的next指针互相指向，形成环形链表，CPU请求直接干烧。

Java8采取了尾插法，但依旧没能解决节点覆盖的问题。

### **扩容为啥要重新插每个节点**？

因为扩容后，key经过哈希计算后在进行取模可能会得到不同的下标，每个桶里面的节点都可能去新的桶里面



### 扩容触发条件(两种)

1. 当负载因子大于0.75时

2. 当数组长度小于64且链表长度大于8时

### 为什么是2倍

可以将取模运算优化为位运算，hash % length 优化为 hash & (length - 1)，只有length是2的幂次时等价，扩2倍保证永远是2的幂次

### ConcurrentHashMap: CAS（compare and swap） + synchronized分别用在什么场景

CAS用在冲突较少时，synchronized用在冲突较多时



### ArrayList和LinkedList底层分别是什么，各自什么场景用

ArrayList的底层是动态扩容的数组，每次扩大1.5倍,扩容是通过浅拷贝进行扩容的，复制的是数组的地址。在读多，写少的情况下比较适合用。LinkedList是用指针来指向其他节点的链表。适合写多读少的情况用，中间插入理论上O(1)，但找位置还是O(n)，实际上总代价O(n)

### fail-fast是什么，怎么触发，怎么避免

fail-fast是迭代器里面用来判断遍历时是否列表有被修改，如果有就直接抛ConcurrentModificationException，程序直接崩，没有就继续。通过的是一个记录被修改次数的变量modCount，还有另外一个变量expectedModCount来比对。通过用迭代器里面的自带的操作可以避免。

### 普通for循环遍历删除有什么坑

删除后后面的元素index前移，下一次循环跳过了，会导致漏掉一些元素



### **Thread六种状态，BLOCKED和WAITING的区别**

NEW, RUNNABLE, WAITING, BLOCKED, TIMED_WAITING, TERMINATED. BLOCKED和WAITING的区别是一个是被动的等待，一个是主动的等待notify()唤醒。

### **synchronized锁的是什么**

实例方法：锁的是this（当前对象实例），静态方法：锁的是Class对象，代码块：锁的是括号里的对象

### **可重入是什么，为什么需要可重入**

可重入是同一个线程继续用同一个锁，如果不能重入就会导致线程把自己锁死。Monitor内部有计数器，重入+1，退出-1，归零才释放锁。

### **可见性问题是什么，为什么会出现**

cpu有缓存，工作内存，修改后不一定立刻同步，所以一个线程修改了变量，另一个线程可能永远看不到

### **volatile解决了哪两个问题，不能解决什么**

可见性问题，禁止指令重排序，不能解决原子性问题

### **ThreadLocal是什么，内存泄漏怎么产生，怎么解决**

是每个线程单独存一份数据，线程之间相互隔离，线程池里的线程不可销毁，value一直被ThreadLocalMap引用着，无法GC回收，用.remove()解决

### **happens-before是什么，说出两条规则**

JVM规定的线程之间可见性和有序性规则。传递性规则，volatile规则，监视器锁规则，程序顺序规则



1. 栈：每线程独享，存局部变量+操作数栈+返回地址，爆了→StackOverflowError
2. 堆：线程共享，存对象实例+实例字段，无引用对象等GC回收
3. 方法区：线程共享，存类信息+静态变量+常量池+方法字节码
4. 永久代→元空间：从JVM固定内存→本地内存，上限从固定变动态
5. 局部变量在栈，实例字段在堆，静态变量在方法区



1. 标记-清除：标记存活→清除垃圾，有内存碎片，可能导致OOM
2. 复制算法：内存一分为二，存活对象复制到另一半，无碎片但利用率50%
3. 标记-整理：存活对象推到一端，整体清空另一端，无碎片+利用率100%但最慢
4. 年轻代用复制：存活率低，复制开销小，无碎片

## 事务隔离级别 + MVCC

### 三种并发问题

- 脏读：读到未提交的数据，对方回滚后数据从未存在
- 不可重复读：同一行读两次不一样，原因是UPDATE
- 幻读：同一范围查两次行数不一样，原因是INSERT/DELETE

### 四个隔离级别

| 隔离级别             | 脏读    | 不可重复读 | 幻读                           |
| -------------------- | ------- | ---------- | ------------------------------ |
| READ UNCOMMITTED(RU) | ❌会发生 | ❌会发生    | ❌会发生                        |
| READ COMMITTED(RC)   | ✅解决   | ❌会发生    | ❌会发生                        |
| REPEATABLE READ(RR)  | ✅解决   | ✅解决      | ⚠️理论上会，MySQL用MVCC基本解决 |
| SERIALIZABLE         | ✅解决   | ✅解决      | ✅解决                          |

### MVCC（Multi-Version Concurrency Control）
在事务开始读数据的那一刻，MySQL会生成一个ReadView。Undo Log存储历史版本，每次修改数据时旧版本写入Undo Log，通过roll_pointer串成版本链。

在去读一行数据时，先看这行数据的 trx_id 是多少

→ 比我的ReadView里最小活跃ID还小？说明这个事务在我开始之前就提交了，我能看到。

→ 在我的活跃列表里？说明这个事务还没提交，我不该看，顺着roll_pointer去找上一个版本。

→ 比下一个事务ID还大？说明这个事务是我开始之后才启动的，我也不该看，继续找上一个版本。

REPEATABLE READ：整个事务只生成一次ReadView，所以每次查看到的世界一致。READ COMMITTED：每次SELECT重新生成ReadView，所以能看到别人刚提交的数据，导致不可重复读。

### 面试答法
**Q1：幻读和不可重复读的区别？**

幻读和不可重复读的区别是前者是对数据进行增删，而后者进行的是修改，这是对已有的数据进行的操作。

**Q2：MVCC怎么实现的？解决了什么问题？**

ReadView中记录了活跃的未提交的事务，这些版本都不该看，通过Undo Log的版本链的roll_pointer指向历史版本，去看上一版。解决了高并发和事务的安全性矛盾问题



- 行锁(记录锁)：锁索引记录，WHERE不走索引 → 全表扫描 → 每行加锁 → 等同表锁
- 间隙锁：锁空隙，防INSERT，解决幻读，RR级别下生效
- 表锁：整张表阻塞，MyISAM默认，InnoDB退化情况出现
- 临键锁(Next-Key Lock) = 记录锁 + 间隙锁。

InnoDB vs MyISAM：
- InnoDB有redo log → 支持崩溃恢复 → 才能真正保证事务原子性
- MyISAM没有redo log → 断电数据损坏 → 不支持事务
- 本质区别不是特性清单，是有没有redo log



**缓存穿透**：黑客可以通过访问大量的不存在的Id，这样每次访问的数据都在redis缓存里找不到，只能每次都跑到数据库里面去找，这就会导致缓存穿透。且因为在数据库里面还是找不到这个数据然后就不会把查找的这个Id存到redis中，以后每次访问这个Id，还是会缓存穿透到数据库，可能会导致数据库炸掉

**解决办法**：

1. 空值+ttl（time to live）。访问的数据如果是一个不存在的数据就把这个空值写入/缓存到Redis(打上短TTL)，并给这个它的寿命设定一个时长，时间一到就把其在Redis中存的给清理掉，以免Redis被占满。但是这个解决办法的坑就是黑客可以通过每次都访问新的Id，来让空值+ttl都没有机会使用出来。

2. 布隆过滤器。每向数据库里面存一个数据，过滤器就会对其进行哈希处理，记录其对应的哈希值。这样可以保证查询的Id进行哈希处理后如果过滤器里面的哈希值没有的对应的话，就说明这个查询的数据是一定不存在的。但是不能保证如果有对应的哈希值的话，就一定有对应的数据。所以真实生产环境中都是通过跟解决办法一一起来处理缓存穿透的问题的。

   布隆过滤器的误判其实是因为不同key经过hash计算后，可能得到相同的hash值，还有布隆过滤器里面存的其实是一个bit数组，存一个key时，转化为hash值时，算出了几个位置就把那几个位置从0置为1，进来查询的只要在那几个中有一个为0，就一定不存在。

   还有预热，在服务启动前，会先把所有数据先往过滤器里面过一遍，不然它都不知道数据库里面有哪些数据。布隆过滤器中不能删除，删除可能导致误伤，因为删除需要把bit位从1置0，但一个bit位可能是多个数据公用的。



**缓存雪崩**：

1. 就是redis突然一下子崩掉了，导致大量的访问全部进入数据库，导致数据库卡死崩掉。
2. redis的大量的key的ttl同时到期，导致大量请求进入数据库，数据库崩掉。

解决办法：

第二种问题的解决办法就是给每个key的ttl设置的不一样，通过基础ttl + 一个随机值（TTL 加随机抖动），这样就不会同一时间大量key 的ttl同时到期。

第一种问题的解决不能跟第一种情况一样，ttl存在redis内存中，跟着redis一起死掉了，所以不可能用ttl来解决。

首先采用从源头根治，第一层：高可用，主从加哨兵，主节点挂了从节点就顶上去。

其次通过限流降级，将mysql允许访问的数量降低，牺牲用户的体验换来数据库的安全。

最后，重建加锁：redis修复好后，重建加锁可以做到不让重复的请求进入数据库，因为这些请求查的是【同一条数据】，所以只放一条请求进来，重建redis缓存。



这三个，它们各管一段，高可用=防Redis挂(治本);限流降级=Redis真挂了那段时间保MySQL(急救);重建加锁=Redis恢复后缓存空、重建那一瞬防重复请求二次冲垮(去重)。

数据库为啥会被冲跨？大量请求同时进来，每个请求被分到的cpu计算时间就变短了，磁盘IO就排长队，每条查询就更慢，导致一条请求连接被一直霸占，请求就一直堆积，磁盘IO排的队就更长，最后MySQL假死或者被OOM killer杀掉。MySQL的服务能力彻底失效，但是数据还在。





**RDB(快照):**

- 文件小、恢复快(直接加载快照,不用重放命令)。
- 致命缺点:**它是"定时拍照"的**。比如每 5 分钟拍一次。如果在第 4 分钟宕机,**最近 4 分钟的数据全丢**(还没来得及拍下一张照)。
- 适合:能容忍丢几分钟数据的场景(缓存、不那么关键的数据)。

**AOF(命令日志):**

- 每条写命令都追加记录(几乎实时),宕机最多丢**一两秒**的数据,**安全得多**。
- 缺点:文件大、恢复慢(要把上亿条命令**重新执行一遍**才能恢复)。
- 适合:对数据丢失敏感、不能容忍丢几分钟的场景。

**持久化**:Redis 数据在内存,宕机清零,需备份到硬盘。 **RDB**:存快照(最终状态)。文件小、恢复快(直接加载);缺点定时拍照,宕机丢最近几分钟数据。 **AOF**:存写命令流水账。几乎实时、最多丢一两秒,安全;缺点文件大、恢复慢(重放所有命令)。 **AOF 重写**:照着内存当前状态,反推最精简命令重写新文件(不是逐条删旧),治文件膨胀;重写期间新命令进缓冲区,完成后补到末尾,不丢数据。 **权衡**:RDB 拿安全换性能,AOF 拿性能换安全。生产常两者并用。
