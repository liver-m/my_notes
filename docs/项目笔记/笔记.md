### GC
是什么
在单增的过程中是young gen生成的过程，那个断崖式下跌就是GC触发把所有young gen都清理掉了
为什么需要
堆内存是有限制的，满了就会卡住，所以需要把一些用不到的垃圾给清理掉
坑点
在GC过程中会触发STW，整个程序会停止执行，这个过程的快慢往往决定着程序运行速度(这里面的理解核心其实要了解堆，它会扫描堆，在堆里但没人引用的对象会被清理掉，更深一步去了解就是可达性分析了（Reachability Analysis）)

### exception 处理
是什么 
将具体是什么错误正确显示在控制台和客户端 
为什么需要 
把服务器内部异常翻译成客户端能理解的 HTTP 语义 
坑点
异常要继承 RuntimeException，否则每个方法都要显式声明 throws，代码很脏

### **哈希函数为什么**O(1)?

经过哈希函数的计算，再进行取模，原来的key会变为在一定范围内的整数，而且这个整数经过这个函数计算并取模后是确定的，因此可以用其作为下标，所以查询速度接近O（1）

### **哈希冲突**？

为了解决多个key经过计算并取模后得到相同的结果，最后会在同一个桶里面，这里采取链表来解决，相同的就继续挂在后面

### **链表太长**？

当数组长度小于64且链表长度大于8时会先采取数组扩容的方法，来缩短链表的长度，如果数组大于等于64且链表长度大于8时，链表还是太长就采取红黑树代替链表

### **线程不安全**？

当扩容数组时，Java7采取头插法进行重新排列链表，多个线程同时操作，这里就有可能出现，节点的next指针互相指向，形成环形链表，CPU请求直接干烧。

Java8采取了尾插法，但依旧没能解决节点覆盖的问题。

### **扩容为啥要重新插每个节点**？

因为扩容后，key经过哈希计算后在进行取模可能会得到不同的下标，每个桶里面的节点都可能去新的桶里面



### 扩容触发条件(两种)

1. 当负载因子大于0.75时

2. 当数组长度小于64且链表长度大于8时

### 为什么是2倍

可以将取模运算优化为位运算，hash % length 优化为 hash & (length - 1)，只有length是2的幂次时等价，扩2倍保证永远是2的幂次

### ConcurrentHashMap: CAS（compare and swap） + synchronized分别用在什么场景

CAS用在冲突较少时，synchronized用在冲突较多时



### ArrayList和LinkedList底层分别是什么，各自什么场景用

ArrayList的底层是动态扩容的数组，每次扩大1.5倍,扩容是通过浅拷贝进行扩容的，复制的是数组的地址。在读多，写少的情况下比较适合用。LinkedList是用指针来指向其他节点的链表。适合写多读少的情况用，中间插入理论上O(1)，但找位置还是O(n)，实际上总代价O(n)

### fail-fast是什么，怎么触发，怎么避免

fail-fast是迭代器里面用来判断遍历时是否列表有被修改，如果有就直接抛ConcurrentModificationException，程序直接崩，没有就继续。通过的是一个记录被修改次数的变量modCount，还有另外一个变量expectedModCount来比对。通过用迭代器里面的自带的操作可以避免。

### 普通for循环遍历删除有什么坑

删除后后面的元素index前移，下一次循环跳过了，会导致漏掉一些元素



### **Thread六种状态，BLOCKED和WAITING的区别**

NEW, RUNNABLE, WAITING, BLOCKED, TIMED_WAITING, TERMINATED. BLOCKED和WAITING的区别是一个是被动的等待，一个是主动的等待notify()唤醒。

### **synchronized锁的是什么**

实例方法：锁的是this（当前对象实例），静态方法：锁的是Class对象，代码块：锁的是括号里的对象

### **可重入是什么，为什么需要可重入**

可重入是同一个线程继续用同一个锁，如果不能重入就会导致线程把自己锁死。Monitor内部有计数器，重入+1，退出-1，归零才释放锁。

### **可见性问题是什么，为什么会出现**

cpu有缓存，工作内存，修改后不一定立刻同步，所以一个线程修改了变量，另一个线程可能永远看不到

### **volatile解决了哪两个问题，不能解决什么**

可见性问题，禁止指令重排序，不能解决原子性问题

### **ThreadLocal是什么，内存泄漏怎么产生，怎么解决**

是每个线程单独存一份数据，线程之间相互隔离，线程池里的线程不可销毁，value一直被ThreadLocalMap引用着，无法GC回收，用.remove()解决

### **happens-before是什么，说出两条规则**

JVM规定的线程之间可见性和有序性规则。传递性规则，volatile规则，监视器锁规则，程序顺序规则



1. 栈：每线程独享，存局部变量+操作数栈+返回地址，爆了→StackOverflowError
2. 堆：线程共享，存对象实例+实例字段，无引用对象等GC回收
3. 方法区：线程共享，存类信息+静态变量+常量池+方法字节码
4. 永久代→元空间：从JVM固定内存→本地内存，上限从固定变动态
5. 局部变量在栈，实例字段在堆，静态变量在方法区



1. 标记-清除：标记存活→清除垃圾，有内存碎片，可能导致OOM
2. 复制算法：内存一分为二，存活对象复制到另一半，无碎片但利用率50%
3. 标记-整理：存活对象推到一端，整体清空另一端，无碎片+利用率100%但最慢
4. 年轻代用复制：存活率低，复制开销小，无碎片
5. 老年代用标记-整理：存活率高，需要100%利用率+无碎片防大对象OOM





### **类加载5步流程（准备 vs 初始化的区别）**

loading -> verification -> preparation -> resolution -> initialization

preparation和initialization的区别是,前者是为为类的静态变量分配内存，并赋默认值（int→0，String→null，boolean→false），后者初始化是赋值

### **三个 ClassLoader 各管什么**

BootstrapClassLoader管的是jdk核心

ExtClassLoader管的是jdk扩展库，javax.*这类

AppClassLoader管的是自己写的代码

### **双亲委派完整流程（一步步）**

收到了任务，自己先不完成，先往父加载器上传，先一直传到最顶端，最大的父加载器要是认识这个类就加载，要是不认识，才往下退，退到谁认识谁就加载，然后把结果原路返回。如果所有父加载器都不认识，最终退回AppClassLoader，由AppClassLoader加载。

### **双亲委派的核心作用**

防止jdk核心库被篡改，优先使用jdk核心库再使用自己定义的类

### **怎么打破**

继承ClassLoader+重写loadClass方法，去掉里面委派父加载器的逻辑



### **IoC(inversion of control)解决了什么问题**

IoC把对象的创建和注入控制权交给了容器，Service只声明需要什么，具体需要塞谁由容器决定——所以依赖可以替换，耦合问题解决了。

###  **Bean生命周期5步**

1. 创建实例：通过构造器先创建一个对象
2. 注入：@Autowired把依赖塞进来
3. 初始化：@PostConstruct执行，在注入完成后做的准备工作
4. 使用：业务代码在跑
5. 销毁：@PreDestroy执行，释放自己持有的资源

### **循环依赖怎么解决**

先把对象创建出来，先不注入字段，对象实例化完立刻存起来，注入阶段需要它时先把半成品交出去，等双方都注入完，半成品自然变成完整品。这个机制叫**三级缓存**。



### **AOP解决什么问题（一句话）**

公共逻辑（事务/日志/权限校验）散落在每个方法里，改需求就改每个方法。AOP把这些逻辑抽到切面，业务代码零侵入。

### **JDK代理 vs CGLIB区别（一张表）**

|          | JDK代理        | CGLIB              |
| -------- | -------------- | ------------------ |
| 怎么伪装 | 实现同一个接口 | 继承原始类生成子类 |
| 要求     | 必须有接口     | 不需要接口         |
| 限制     | 没接口歇菜     | `final`类歇菜      |

### **`@Transactional`失效的场景和原因**

同类内部用`this`调用，绕过代理对象，AOP失效，`@Transactional`不生效.
